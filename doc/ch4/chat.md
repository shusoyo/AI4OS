### 🤖 Assistant

这是一个非常深刻的问题。要理解“传送门”机制为什么能支撑切换而不崩溃，我们需要从 CPU 执行指令的底层逻辑——**“地址翻译的连续性”**切入。
你可以试着从以下几个关键点来推导它的工作原理：

### 1. 核心矛盾：地址翻译的“瞬间失联”
在开启分页机制后，CPU 发出的每一个指令地址（PC）都是**虚拟地址**。

- 当你修改 `satp` 寄存器（切换页表）的那一瞬间，CPU 还在执行下一条指令。
- 如果旧页表和新页表在当前 PC 所指向的那个虚拟地址映射不同（或者新页表干脆没映射），CPU 就会因为找不到下一条指令而立即触发异常或崩溃。

### 2. 传送门的“幻术”：虚拟地址的完全重合
为了解决这个问题，内核在构建地址空间时做了一个特殊处理：

- **物理层面**：分配一个专门的物理页，里面存放着切换 `satp` 和保存/恢复寄存器的汇编代码。
- **逻辑层面**：在内核页表和**所有**用户页表中，都把这个物理页映射到**同一个虚拟地址**（在代码中是 `VPN::MAX`，即虚拟地址空间的最后一页）。

### 3. 切换瞬间发生了什么？
想象你正在一个房间（内核空间）里，通过一扇门（传送门代码）。

1. **进入门内**：PC 跳转到 `VPN::MAX` 对应的某个位置。
2. **执行切换**：指令修改了 `satp`。
3. **时空转换**：此时地址翻译瞬间切换到了新页表（用户空间）。
4. **依然在门内**：**关键点来了**——虽然页表换了，但新页表在 `VPN::MAX` 这个地址映射的依然是刚才那个物理页，内容一模一样。
5. **继续执行**：对于 CPU 来说，它发现 `satp` 变了，但当前执行的指令流在虚拟地址上依然是连续的，代码依然在那里。

### 2. 算术题：地址与页号的转换
在代码里，我们看到映射的范围是 `VPN::new((1 << 26) - pages)..VPN::new(1 << 26)`。

- **计算公式**：$\text{虚拟地址} = \text{虚拟页号} \times \text{页面大小} (2^{12})$。

$$
\text{目标地址位 (第 38 位)} - \text{页偏移位 (12 位)} = \text{VPN 的第 26 位}
$$

### 第一部分：为什么 `stvec` 会指向“无效映射”？

#### 1. 核心背景：`stvec` 存的是什么？
在 RISC-V 中，`stvec` 寄存器存放的是一个**虚拟地址**。当 CPU 遇到异常（如访存失败）时，它会查当前的页表，把 `stvec` 里的虚拟地址翻译成物理地址，然后跳过去执行。

#### 2. 灾难现场模拟
想象一下 `execute_naked` 汇编执行到这一行：

```代码段
" csrw satp, {user_satp} " // 假设这是切换页表的指令

```

- **瞬时切换**：一旦这条指令执行完，CPU 瞬间就进入了**用户地址空间**。此时，所有的地址翻译都必须由用户页表说了算。
- **异常触发**：由于你提到的“忘记映射传送门”，CPU 在执行下一条指令时发现当前 PC（程序计数器）指向的地址在用户页表里是**无效**的（$V=0$）。于是，CPU 立即触发 **Instruction Page Fault**。
- **寻找救命稻草**：CPU 按照硬件逻辑，低头去读 `stvec` 寄存器。
- **逻辑闭环**：在 `LocalContext` 的实现中，`stvec` 被设置成了标签 `1:` 的地址。**关键点来了**：标签 `1:` 和刚才切换 `satp` 的那行代码，通常是在**同一个物理页面**（即传送门页面）内。

#### 3. 为什么是“无效映射”？
因为你“忘记映射”了整个传送门页面，这意味着：

1. 当前代码所在的页面没映射。
2. `stvec` 指向的那个 `1:` 所在的页面，**由于是同一个页面，自然也没映射**。

#### 1. 经典内存布局：三明治模型
Linux 采用了“两头往中间长”的策略，将虚拟地址空间分成了几个清晰的区域：

- **低地址区**：存放程序的代码段（Text）、数据段（Data）和 BSS 段。
- **堆（Heap）**：紧跟在 BSS 后面，通过 `brk` 指令往**高地址**长。
- **栈（Stack）**：位于用户空间的最高端附近，往**低地址**长。
- **内存映射区（Memory Mapping Segment）**：这就是 `mmap` 的家。它通常位于堆和栈之间的巨大空隙里。

#### 2. “安全地址”是怎么找的？
当你调用 `mmap` 且 `addr` 传 0（让 OS 自己找）时，内核会执行以下逻辑：

1. **确定起始基准**：Linux 有一个 `TASK_UNMAPPED_BASE`（通常是 $2^{31}$ 或更高），作为 `mmap` 的起始搜索点。
2. **管理 VMA 链表**：内核通过 `vm_area_struct`（VMA）结构体记录了所有**已经映射**的区间。
3. **寻找“空洞” (Gap)**：
  - 内核会遍历 VMA 链表，寻找两个已存在区间之间长度 $\ge len$ 的**空闲区域**。
  - **算法优化**：为了效率，它不是乱找。它通常采用“自顶向下”或“自底向上”的搜索策略，并利用 **红黑树 (Red-Black Tree)** 快速定位不冲突的空洞。
4. **预留缓冲区（Gap）**：为了防止堆长着长着撞上 `mmap`，或者 `mmap` 撞上栈，内核在它们之间会强制留出一段 **Guard Page**（保护页）。

#### 3. 避开堆和栈的绝招

- **动态调整**：如果堆区（`brk`）长到了 `mmap` 区域的边界，`sbrk` 会返回失败（内存不足）。
- **地址空间随机化 (ASLR)**：为了安全，Linux 每次启动程序时，`mmap` 的基地址、堆的起点、栈的起点都会随机偏移一段距离，让黑客无法预判地址。
